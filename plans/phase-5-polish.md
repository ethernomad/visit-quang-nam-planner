## Resilience tasks

### 1. Loading state

- Replace the Phase 4 generic spinner with a skeleton: shimmer
  placeholder cards matching the day-card and timeline layout. Use
  Tailwind's `animate-pulse` or a custom `@keyframes shimmer` in
  `input.css`.
- Show a timeout hint at the 8-second mark: "Taking a little longer
  than usual — the model is thinking." Use a `use_effect` that
  triggers off `resource.state() == Pending` + a `tokio::time::sleep`
  guarded by a `use_signal` flag (do not block rendering).
- Cap the wait at 60 s in the client (a Dioxus `use_resource` task
  that aborts after 60 s) — the server (Phase 3) should also enforce
  this via `async-openai`'s request timeout, so a hung client is the
  backstop, not the only safeguard.

### 2. Error state

Promote the Phase 4 error block to `ErrorBox`:

```rust
#[component]
fn ErrorBox(error: ServerFnError, on_retry: Callback<()>) -> Element {
    let (title, body) = classify_error(&error);
    rsx! {
        div { class: "...",
            h3 { "{title}" }
            p { "{body}" }
            button { onclick: move |_| on_retry.call(()), "Try again" }
        }
    }
}

fn classify_error(e: &ServerFnError) -> (&'static str, String) {
    match e {
        ServerFnError::ServerError { code: 503, .. } => ("The planner is taking a break", "OpenAI returned 503; wait a moment and try again.".into()),
        ServerFnError::ServerError { message, .. } if message.contains("OPENAI_API_KEY") => ("Server config error", "The server has no OpenAI key. This is a misconfiguration.".into()),
        ServerFnError::ServerError { message, .. } if message.contains("no grounding chunks") => ("We don't have enough content for that trip", "Try widening your interests.".into()),
        _ => ("Something went wrong", e.to_string()),
    }
}
```

Categorising errors is the single highest-leverage UX change here — a
panic-y "anyhow error string" makes every server hiccup look like a
bug.

  lodging (+30), local food (+20), low-carbon transport (+15), off-peak
  timing (+17) = 82/100". This needs Phase 3 to populate a small

### 4. Empty / edge cases

- **Zero activities on a day** — render a row "No activities planned
  for this day. Try adjusting your interests." Don't render an empty
  timeline.
- **`source_url` missing** on an activity — render the row without a
  "Read more" link rather than crashing. Already mandated by Phase 4;
  add a quick unit test to lock it in.
- **Duplicate activities across days** — rare, but if `itinerary.days`
  has the same `title` on two different days, log from the client
  (don't hide — diagnostics help Phase 3). Don't try to deduplicate
  in the UI; that's the server's job.

## Notes for the agent

- This phase is the user's first real click-through surface; spend
  extra cycles on the loaded-and-loaded-errored loop, not new fields.
  Adding inputs here will undo the locked-schema discipline from
  Phase 3.
- If you find yourself needing a new domain field (e.g.
  `sustainability_breakdown`), make it an **option** on the type (Vec
  defaults to empty) so Phase 3 responses without the field still
  parse. Backwards compat.
- Don't introduce `dioxus-i18n` yet. Leaving copy in plain Rust consts
  keeps the swap mechanical for Phase 6; introducing the fluent bundle
  mid-polish complicates every component file.
- The timeout/backstop "fix this locally in the client" approach is
  Gun Safety 101 — if you're tempted to fix it server-side instead,
  do, but document the change in `plans/phase-3-llm.md` so thePhase 3
  plan doesn't drift from reality.
- For accessibility keyboard nav, prefer `tabindex`/`onkeydown` over
  raw JS hacks. Dioxus 0.7 events expose `key: Key` (Arrow codes work).
